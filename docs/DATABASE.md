# Tipon — Database Schema & Frontend Usage Reference

This document covers all 4 tables in the Tipon database: what each column is for,
the design decisions behind them, and exactly how `tipon-web` (the React frontend)
consumes each one. Written to answer supervisor/reviewer questions about the ERD
without having to dig through migrations and components live.

Source files referenced throughout:
- Migrations: `tipon-api/database/migrations/`
- Models: `tipon-api/app/Models/`
- Frontend API types + adapters: `tipon-web/src/app/lib/api.ts`,
  `tipon-web/src/app/store/AppStore.tsx`

**A third consumer now exists in the current application:** the participant
Livewire pages read and write the `events`, `registrations`, and `notifications`
tables through Laravel/Eloquent and normal web routes. Their classes live in
`tipon-api/app/Livewire`, their component views live in
`tipon-api/resources/views/livewire`, and their page wrappers live in
`tipon-api/resources/views/events`. Unlike `tipon-web`, these pages consume the
schema directly instead of adapting API responses into camelCase frontend types.
Noted per-column below where relevant.

---

## 1. `users`

Single table for all account types — participants, organizers, and admins — 
distinguished only by the `role` column. There are no separate tables per role.

| Column | Type | Purpose |
|---|---|---|
| `id` | bigint (PK, auto-increment) | Unique user identifier. Referenced by `events.organizer_id`, `registrations.user_id`, `notifications.notifiable_id`. |
| `name` | string | Display name shown throughout the UI. |
| `email` | string, **unique** | Login identifier. Uniqueness enforced at the DB level. |
| `password` | string | Bcrypt hash (`'password' => 'hashed'` cast in `User.php` — hashed automatically on set). Hidden from all API responses (`$hidden`). The raw value must satisfy a strength policy (added 2026-07-02) — min 8 characters, mixed case, a number, and a symbol — enforced via Laravel's `Password::min(8)->mixedCase()->numbers()->symbols()` rule on both `AuthController::register` and `AdminController::createOrganizer`. Only the already-validated plaintext ever reaches this column; the requirement itself isn't stored anywhere, it's re-checked on every write. |
| `role` | enum: `participant` \| `organizer` \| `admin`, default `participant` | Drives all route/UI access control. `admin` was added later via a migration that altered the Postgres check constraint (`2026_07_01_120000_add_admin_role_to_users_table.php`). |
| `email_verified_at` | timestamp, nullable | **Now actively used** (added 2026-07-02): `null` until the user confirms a 6-digit OTP sent to their email. `AuthController::login` blocks with `403` while this is null. Set immediately at creation for admin-created organizer accounts and the seeded admin, since they don't go through self-registration. |
| `email_verification_code` | string, nullable | Bcrypt-hashed OTP code (never stored/compared in plaintext — checked via `Hash::check()`), generated by `AuthController::issueVerificationCode()`. Cleared back to `null` once verified. Hidden from all API responses (`$hidden`). |
| `email_verification_code_expires_at` | timestamp, nullable | Code is valid for 10 minutes from issuance. `verifyEmail()` rejects an expired code even if it otherwise matches. Hidden from all API responses. |
| `remember_token` | string, nullable | Supports cookie-based "remember me" logins. Tipon uses Sanctum token auth instead — **unused boilerplate** from Laravel's default scaffolding. |
| `created_at` / `updated_at` | timestamp | Standard record-keeping. |

**Relationships:** `organizedEvents()` (has many `Event` via `organizer_id`), `registrations()` (has many `Registration`).

**Email verification flow:** `POST /register` creates the account unverified and emails a code (no token issued yet) → `POST /email/verify` (email + code) checks the hash and expiry, sets `email_verified_at`, clears the code columns, and issues the Sanctum token → `POST /email/resend` (rate-limited to 3/min) issues a fresh code for an unverified account. Mail is sent via Brevo's HTTP API transport (not SMTP), registered through `Mail::extend('brevo', ...)` in `AppServiceProvider::boot()` — this was necessary because the dev machine's ISP blocks outbound SMTP ports entirely.

### Frontend usage (`tipon-web`)

| Column | Frontend field | Usage |
|---|---|---|
| `id` | `User.id` | Keys, identifies `currentUser`, filters "my registrations", path param for `PUT /admin/users/{id}/promote`. |
| `name` | `User.name` | Avatar initials + name everywhere a person is displayed (registrant list, admin table, event organizer attribution). |
| `email` | `User.email` | Admin user table, registrant list, `AuthPage` login field. Note: when a user is discovered indirectly via an event's nested `organizer` relation, `email` is filled in as `""` since that response shape omits it. |
| `role` | `User.role` | The most consequential field client-side: picks which dashboard/routes render, whether the "Promote" button shows (`AdminDashboard.tsx`, only for `participant` rows), and whether `loadAppData()` fetches `myRegistrations` or `organizerRegistrations`. |
| `email_verified_at` | *(unused directly)* | Fetched into `ApiUser` type but never read by name — its *effect* (blocking login, triggering the OTP screen) is driven by the `403`/`requires_verification` response from `/login`, not by reading this column client-side. |
| `created_at` | shown raw (no adapter) | Rendered as "Joined" date in the admin user table. |
| `updated_at` | *(unused)* | Never referenced. |
| `password` / `remember_token` / `email_verification_code` / `email_verification_code_expires_at` | *(never sent)* | All hidden server-side (`$hidden` on `User.php`); the frontend never receives them and has no notion of the OTP value itself — only the email address and the code the user types in. `password` is the one exception worth noting: the *policy* (not the column) is mirrored client-side — `AuthPage.tsx`'s `RegisterForm` re-implements the same 5 rules (length, uppercase, lowercase, number, symbol) as a live checklist and strength bar purely from the in-memory input value, so the user sees the requirement before submitting rather than discovering it from a `422`. The backend re-validates independently regardless — the frontend check is UX only, not the security boundary. |

Two adaptation paths: `adaptUser()` builds `currentUser` (from `/me`, `/login`,
`/register`); the admin table consumes raw `ApiUser[]` from `/admin/users` directly.
The store also opportunistically back-fills a lightweight `users` list from
`organizer`/`user` objects nested in event/registration responses (`mergeUsers`).

## 2. `events`

| Column | Type | Purpose |
|---|---|---|
| `id` | bigint (PK, auto-increment) | Unique event identifier; referenced by `registrations.event_id`. |
| `organizer_id` | bigint (FK → `users.id`), `cascadeOnDelete` | Owning organizer. Deleting the user deletes their events too (DB-enforced). |
| `title` | string | Primary label everywhere the event is shown. Capped at **100 characters** (added 2026-07-02) via `max:100` in `EventController::store`/`update`'s validation — application-level only, not a DB column-length constraint (the column itself is still Laravel's default `VARCHAR(255)`, since nothing else writes to `title` outside those two validated endpoints). Must also be unique (case-insensitive) among that same organizer's other **active** events — see the partial unique index below. Two different organizers may still both use the same title. |
| `description` | text, nullable | Longer free-text detail, shown on the event detail page. Capped at **1000 characters** (added 2026-07-02) via `max:1000` in `EventController::store`/`update`'s validation — same as `title`, this is application-level only; the DB column type (`text`) has no practical length ceiling of its own. Before this, the field had no length limit at all, which was a real gap (unbounded input is a minor storage/DoS vector, and the UI already truncates with `line-clamp-2` regardless of how long the text is). |
| `venue` | string | Location, always required. |
| `event_date` | timestamp | Cast to PHP `datetime` server-side (`Event.php`) for comparisons/formatting. |
| `capacity` | unsigned integer | Max registrants. Ceiling checked inside the transaction-locked registration flow (FR-17). |
| `status` | enum: `open` \| `cancelled`, default `open` | Whether registrations are accepted. **No `full` status** — fullness is always derived from live registration count vs. capacity, never stored, so it can't drift out of sync. |
| `cover_image_path` | string, nullable | Path/URL to an uploaded cover image (`Storage::disk('public')` + `storage:link`). Stores a reference, not a BLOB. The upload itself (added 2026-07-02) is restricted to **JPG/PNG/WEBP, max 2 MB** via `UploadController`'s validation rule — `['image', 'mimes:jpg,jpeg,png,webp', 'max:2048']`. `image` confirms the file is an actually-decodable image (not just a renamed extension); `mimes` narrows the whitelist to exclude SVG (can carry embedded scripts/XXE) and GIF. This column itself never stores or enforces the policy — it's just the resulting path once a valid upload succeeds. **The path also never contains the organizer's original filename** — `Storage::store('covers', 'public')` is called without an explicit target name, so Laravel auto-generates a random hashed one (e.g. `covers/K3sdCdxVjYN2HrDRcoHXrNDWhaIW4VdvxrUmoExG.png`) and the original name the browser sent is discarded entirely. Prevents path traversal via a crafted filename, cross-organizer filename collisions, and leaking whatever the original filename contained. |
| `created_at` / `updated_at` | timestamp | Standard record-keeping. |

**Relationships:** `organizer()` (belongs to `User`), `registrations()` (has many `Registration`).

**Design notes:** cancelling an event flips `status` rather than deleting the row
(preserves history); capacity fill is always computed live from `registrations`.

**Duplicate-title prevention (added 2026-07-02):** a partial unique index —
```sql
CREATE UNIQUE INDEX events_organizer_title_unique
ON events (organizer_id, LOWER(title))
WHERE status = 'open'
```
— blocks an organizer from having two **active** events with the same title,
case-insensitively. The `WHERE status = 'open'` clause means cancelling an event
frees its title for reuse (e.g. cancel "Job Fair 2026" and recreate it under the
same name for a rescheduled date), and different organizers can independently use
the same title since the index is scoped per `organizer_id`. Same pattern as the
`registrations_event_user_unique` partial index — DB-level enforcement backstops
the friendly `422` check already done in `EventController::store()`/`update()`
(`hasDuplicateTitle()`), so the rule holds even under a concurrent double-submit.

### Frontend usage (`tipon-web`)

| Column | Frontend field | Usage |
|---|---|---|
| `id` | `EventItem.id` | Route params, keys, `eventById()`/`confirmedCountFor()` lookups, FK when creating a registration. |
| `organizer_id` | `EventItem.organizerId` | Filters "my events" on the organizer dashboard. |
| `title` | `EventItem.title` | Card/detail-page headline, bar-chart labels (truncated to 16 chars). The Title `<Input>` in `EventFormDialog` has `maxLength={100}` (browser-enforced) plus a live `X/100` character counter next to the label that turns red at the limit — mirrors the backend's `max:100` rule the same way the duplicate-title check mirrors the partial unique index. `EventFormDialog` also compares the in-progress title against the already-loaded `events` list (same organizer, `status === "open"`, excluding the event being edited) to show a live duplicate warning and disable submit — mirrors the DB-level rule client-side for instant feedback, same UX pattern as the password-strength checklist. |
| `description` | `EventItem.description` (`""` if null) | `EventCard` (`line-clamp-2`) and detail page. The `<Textarea>` in `EventFormDialog` has `maxLength={1000}` plus a live `X/1000` character counter, same treatment as `title`'s `X/100` counter. |
| `venue` | `EventItem.venue` | Shown with a map-pin icon. |
| `event_date` | `EventItem.eventDate` | Formatted via `formatEventDate()`/`formatEventTime()`; drives Upcoming/Past sorting on `MyRegistrations`. |
| `capacity` | `EventItem.capacity` | Feeds `<CapacityBar>`, `remainingFor()`, `isFull()`. |
| `status` | `EventItem.status` | `EventStatusBadge`, hides the capacity bar when cancelled, gates registration/edit actions. |
| `cover_image_path` | `EventItem.cover_image_path` (`""` if null) | `<ImageWithFallback>` on cards/detail; the field the upload pipeline writes into. `EventFormDialog`'s file picker mirrors the backend policy client-side — `accept="image/jpeg,image/png,image/webp"` on the `<input>`, plus a pre-upload check rejecting wrong types/oversized files (>2 MB) instantly with a toast, before ever calling `POST /upload`. As with the password-strength checklist, this is UX only; `UploadController`'s server-side validation is the actual enforcement. |
| `registered_count` *(API-computed, not a DB column)* | `EventItem.registeredCount` | `confirmedCountFor()` uses it directly if present, else falls back to counting the local `registrations` array — a defensive fallback, not a real schema field. |
| `created_at` / `updated_at` | *(unused)* | Fetched but dropped by `adaptEvent()`; never shown. |

`isFull(eventId)` — used for the "Full" badge on `EventCard` — compares
`confirmedCountFor(eventId)` against `capacity` client-side, mirroring the backend's
derived-capacity design rather than trusting a stored flag.

### Livewire usage

`app/Livewire/EventBrowse.php` queries `Event::query()` directly with `withCount`
for the live registered count, using the same derived-count design as
`EventController::index`. Search is supported through the query string
(`GET /events?q=...`) and filters across `title`, `description`, and `venue`, so
the page remains usable even when Livewire JavaScript hydration is unreliable.

`app/Livewire/EventDetail.php` receives `{event}` through Laravel route model
binding via `EventController@detailPage`, then the Blade wrapper renders
`<livewire:event-detail />`. The component reads `title`, `description`,
`event_date`, `venue`, `organizer->name`, `capacity`, and `status` directly from
the model. The page still uses the same database fields as the React detail page,
but without a JSON API adapter layer.

## 3. `registrations`

Junction table linking `users` (participants) to `events` — the record of who
signed up for what.

| Column | Type | Purpose |
|---|---|---|
| `id` | bigint (PK, auto-increment) | Unique registration identifier; used in `DELETE /registrations/{id}` and `PUT /registrations/{id}/attendance`. |
| `event_id` | bigint (FK → `events.id`), `cascadeOnDelete` | Which event. |
| `user_id` | bigint (FK → `users.id`), `cascadeOnDelete` | Which participant. |
| `status` | enum: `registered` \| `cancelled`, default `registered` | Cancelling flips this rather than deleting the row — preserves history. |
| `attendance` | enum: `pending` \| `present` \| `absent`, default `pending` | Set by the organizer after the event. Independent of `status`. |
| `created_at` / `updated_at` | timestamp | `created_at` = when they registered; `updated_at` changes on `status`/`attendance` updates. |

**The key design detail — a partial unique index:**
```sql
CREATE UNIQUE INDEX registrations_event_user_unique
ON registrations (event_id, user_id)
WHERE status = 'registered'
```
A user can have at most one **active** registration per event, but can re-register
after cancelling — the index only applies while `status = 'registered'`. This makes
duplicate-prevention (FR-16) a database guarantee, not just an app-level check.

**Relationships:** `event()` (belongs to `Event`), `user()` (belongs to `User`).

### Frontend usage (`tipon-web`)

| Column | Frontend field | Usage |
|---|---|---|
| `id` | `Registration.id` | Keys; path param for cancel and `recordAttendance()`. |
| `event_id` | `Registration.eventId` | Links back to its event via `eventById()`; used by `confirmedCountFor()`/`isFull()` to count fill per event. |
| `user_id` | `Registration.userId` | Filters "my registrations"; `registrationFor(eventId)` checks "have I already registered" — drives the "Registered" badge and disables the register button. |
| `status` | `Registration.status` | Core filter for `MyRegistrations`' three tabs (Upcoming / Past / Cancelled); gates `confirmedCountFor()` (only `registered` rows count toward capacity); rendered via `RegistrationStatusBadge`. |
| `attendance` | `Registration.attendance` | `AttendanceBadge`, shown only on the Past tab. On the organizer side: `RegistrantList`'s Present/Absent buttons read/write it; `OrganizerDashboard` aggregates it into the attendance-rate stat and pie chart. |
| `created_at` | `Registration.createdAt` | "Registered" date column in `RegistrantList`'s registrant table. (`MyRegistrations` sorts by the event's date instead, not this column.) |
| `updated_at` | *(unused)* | Not even included in the frontend type. |

Cancelling a registration client-side just flips `status` to `"cancelled"` locally
(mirrors the backend's soft-cancel) — it stays in the `registrations` array and still
shows up under the Cancelled tab.

### Livewire usage

The participant Event Detail page now uses normal Laravel form POST fallbacks for
registration and cancellation:

- `POST /events/{event}/register`
- `POST /events/{event}/cancel-registration`

Those routes call `RegistrationController` page methods and still use the same
transaction-locked capacity check as the API (`Event::lockForUpdate()` before
counting active `registered` rows). Cancellation flips `status` to `cancelled`,
preserving history. `attendance` is never read or written here; attendance
recording remains organizer-only on the React side.

## 4. `notifications`

Laravel's built-in polymorphic notifications table (generated by
`php artisan notifications:table`), not custom-built for Tipon.

| Column | Type | Purpose |
|---|---|---|
| `id` | uuid (PK) | Unguessable notification ID. |
| `type` | string | Fully-qualified PHP class name that generated the row, e.g. `App\Notifications\RegistrationStatusNotification`. Tells Laravel how to interpret `data`. |
| `notifiable_type` | string | The model class this notification belongs to — always `App\Models\User` in Tipon. |
| `notifiable_id` | bigint | The ID of that user. Together with `notifiable_type` this forms a **polymorphic relationship**: the same table could target any model type, not just users, without a schema change — a Laravel convention, not a Tipon-specific design choice. |
| `data` | text (JSON) | The payload. In Tipon: `event_id`, `event_title`, `status` (`registered`/`cancelled`), and a human-readable `message` (see `RegistrationStatusNotification::toDatabase()`). |
| `read_at` | timestamp, nullable | `null` = unread. Set when the user opens/dismisses it. Powers the unread badge. |
| `created_at` / `updated_at` | timestamp | Standard record-keeping. |

**Why polymorphic:** `$request->user()->notifications()` (used in
`NotificationController::index`) only works because of this setup — it's provided by
Laravel's `Notifiable` trait on the `User` model.

### Frontend usage (`tipon-web`)

`adaptNotification()` reshapes each raw row into `AppNotification`:

| Column | Frontend field | Usage |
|---|---|---|
| `id` | `AppNotification.id` | Keys; path param for `PUT /notifications/{id}/read`. |
| `type` | *(unused)* | The Laravel class name is fetched but ignored — display type is derived from `data.status` instead. |
| `notifiable_type` | *(unused)* | Never read; the API already scopes to the current user. |
| `notifiable_id` | `AppNotification.userId` | Client-side filter (`notifications.filter(n => n.userId === currentUser.id)`) — a safety net on top of the API's own scoping. |
| `data.status` | drives `AppNotification.type` (`"registration_confirmed"` \| `"event_cancelled"`) | Picks icon/color and title string in `NotificationBell`. |
| `data.message` | `AppNotification.body` | The notification's descriptive text. |
| `data.event_id`, `data.event_title` | *(fetched, unused)* | Round-tripped but not currently read by the UI — no "jump to event" link is wired up yet. |
| `read_at` | `AppNotification.readAt` | Drives the unread badge count, the unread dot, row highlighting; clicking marks it read (optimistic update + `PUT /notifications/{id}/read`). |
| `created_at` | `AppNotification.createdAt` | Sorts notifications newest-first; renders relative time ("2h ago"). |
| `updated_at` | *(unused)* | Not part of the frontend type at all. |

**Known gap:** the frontend only derives two notification types
(`registration_confirmed`, `event_cancelled`) from `data.status`, even though
`mockData.ts` still defines two more (`event_updated`, `event_reminder`) that the
backend never actually sends — leftover prototype types, not live features.

### Livewire usage

The Livewire layout reads the authenticated user's latest database notifications
directly through `$user->notifications()`. The unread badge is powered by
`$user->unreadNotifications()->count()`, matching the `read_at = null` convention.

Two session-authenticated web routes update notification read state:

- `POST /notifications/{notification}/read` marks one notification as read.
- `POST /notifications/read-all` marks all unread notifications as read.

The Livewire notification popover follows the same display convention as the React
popover: `data.status = registered` becomes "Registration confirmed", while
`data.status = cancelled` becomes "Registration cancelled". The message body comes
from `data.message`, and `created_at` is displayed as relative time.

## Cross-Table Design Themes

1. **Single table per entity, no per-role duplication** — `users` uses one `role`
   enum instead of separate participant/organizer/admin tables.
2. **Derived values over stored counters** — event fill/fullness and attendance rate
   are always computed live from `registrations`, never cached as a column. Applied
   consistently on the backend (no `filled` column), the React frontend
   (`confirmedCountFor()`, `isFull()`), and independently again in the Livewire
   pages (`registeredCount()`/`isFull()` computed properties) — three separate
   implementations of the same rule, none of them trusting a stored flag.
3. **Soft state changes over hard deletes** — cancelling an event or a registration
   flips a `status` enum rather than deleting the row, preserving history.
4. **Database-enforced integrity** — the partial unique index on `registrations`,
   the equivalent partial unique index on `events` (per-organizer, case-insensitive
   title uniqueness among active events), and the `role` check constraint on
   `users` mean core business rules hold even if application code has a bug, not
   just when the frontend behaves.
5. **Laravel conventions inherited, only some built out** — `remember_token` and
   the polymorphic `notifiable_type`/`notifiable_id` columns are standard Laravel
   scaffolding Tipon left unused (no cookie-session auth, only one notifiable model
   type). `email_verified_at`, by contrast, **is** now fully wired up (as of
   2026-07-02) via a custom OTP flow rather than Laravel's built-in click-link
   verification — worth distinguishing "inherited but unused" from "inherited and
   actively used with custom logic" if asked.
